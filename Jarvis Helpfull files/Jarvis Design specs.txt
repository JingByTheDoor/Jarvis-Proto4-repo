System Pilot Desktop Operator Console: Complete UI and Design Document (Current State)

1. Product Summary
The app is a Windows-first Electron desktop “operator console” for running local automation tasks through a plan/approval/execute workflow.

It is intentionally styled like a modern terminal-control panel:

Dark background.
Monospace typography.
Status chips and telemetry.
Text-first operational output.
Safety gating before risky actions execute.
Primary objective: let an operator preview a plan, approve sensitive actions, execute with live event streaming, and inspect final run artifacts.

2. Platform and Tech

UI: React renderer.
Styling: single CSS theme file.

11. Interaction Model (End-to-End)
11.1 Preview Flow
User enters request.
UI sends preview IPC (pilot:preview) with session id.
UI enforces 20s timeout.
If plan returns:
Plan stored in state.
Risk evaluated.
Active tab shifts to actions if approvals are needed, else plan.
Console logs “preview only” status.
If error:
Error entry added to console.
Floating error toast shown.
11.2 Execute Flow
If current plan missing or stale, UI auto-runs preview first.
UI checks all required approvals.
If any required action undecided/denied:
Execution is blocked.
User is redirected to actions tab.
If allowed:
UI sends execute IPC (pilot:execute) with stream mode.
UI listens for event stream.
Action statuses and logs update live.
On completion:
Run log captured from run_complete.
UI also tries persisted run log file for authoritative final artifact data.
12. Safety and Approval Semantics
requires_approval actions cannot execute without decision.
deny blocks execution.
approve_once satisfies only that action instance.
approve_session creates reusable session-level allowance for matching action signature (type + args).
Pending approvals are counted and shown in bottom bar queue.
Risk meter logic:

Any high-risk action -> DANGER.
Else any medium-risk action -> CAUTION.
Else SAFE.

17. Known UX/Design Gaps in Current Build
Boot screen text implies keypress continue, but code uses timer-only transition.
abort exists in API but no cancel/stop button in UI.
Heavy JSON exposure in rail can be noisy for non-technical operators.
Visual hierarchy is serviceable but could better emphasize “current task + approvals + risk”.
18. How a New User Should Operate It
Launch app and wait for boot panel to pass.
Type a task in command input.
Press Preview.
Open Plan tab to review structure and risk.
Open Actions tab and decide approvals for any gated actions.
Press Execute.
Watch progress and step output in console.
Use Logs tab for final result and artifacts.
Confirm bottom telemetry and queue return to idle-safe state.
